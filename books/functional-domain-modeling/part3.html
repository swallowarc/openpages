第8章

関数の理解

💡

この章を一言で

関数を値と同じ「物」として扱い、合成可能にする。さらに、すべての入力に対して値を返す全域関数を用い、例外を避ける。

🧮関数を「物」として合成する

関数は引数にも戻り値にもできる値であり、つなげて新しい関数を作れる。

著者の筋道は、関数を値として扱えるなら、ある関数の出力を次の関数の入力につなぐことで、小さな関数から大きな処理を組み立てられる、という点にある。これが第7章のパイプライン合成を実装で支える基盤になる。型が合えば、関数は部品のように連結できる。

  • 関数を引数・戻り値として渡し、合成して新しい処理を作る。
  • 出力型と入力型が合う関数どうしを連結する。

✅全域関数で例外を避ける

すべての入力に対して必ず値を返す全域関数を用い、想定外で例外を投げない設計にする。

著者の論拠は、入力の一部で例外を投げる関数（部分関数）は、シグネチャに失敗の可能性が表れず、呼び出し側が見落とす、という点にある。全域関数にすれば、起こりうる失敗も戻り値の型に表れ、扱い忘れを型で防げる。これが第10章の Result 型によるエラー処理につながる。

  • 部分関数（一部の入力で例外を投げる）を避ける。
  • 失敗も戻り値の型に表し、呼び出し側に明示する。
📎
補足（本書外）

全域関数（total function）は、定義域のすべての入力に対して値を返す関数を指す。一部の入力で例外や未定義になる関数は部分関数（partial function）と呼ばれる。

第9章

実装：パイプラインの合成

💡

この章を一言で

関数の型を先に決め、それに沿って各ステップを実装する。依存は関数の引数として注入し、テスト可能にする。

🧭関数の型が実装を導く

先に定義したステップの型をたよりに、各ステップの中身を実装していく。

著者は、入力型と出力型が決まっていれば、実装すべきことの範囲が型によって絞られると説く。検証ステップ、価格計算ステップなどを、それぞれ定められた型に合わせて実装し、最後に合成して一本のパイプラインにする。型が先にあることで、実装が設計から逸れにくい。

  • 検証・価格計算などの各ステップを、定義済みの型に合わせて実装する。
  • 実装したステップを合成し、組み上がったパイプラインにする。

💉依存を関数の引数として注入する

外部への依存をステップの引数として渡し、ステップを純粋に保ってテストしやすくする。

著者の筋道は、依存を関数の中に隠すと、その関数を実行するのに実物の依存が必要になりテストしにくい、という点にある。依存を引数として外から渡せば、テスト時には代わりの実装を渡せる。これにより、ステップ単体をインフラなしで検証できる。

  • 依存を引数として渡し、ステップ内部に隠さない。
  • テスト時は代替の依存を渡し、ステップ単体で検証する。

第10章

実装：エラーの扱い

💡

この章を一言で

例外でなく Result 型でエラーを明示し、成功と失敗の2本の道として扱う。bind と map で関数を連結し、失敗は自動的に失敗の道へ流す。

🎯Result 型でエラーを明示する

失敗を例外で投げるのではなく、戻り値の型（Result）に表して扱い忘れを防ぐ。

著者の論拠は、例外はシグネチャに表れないため、どの関数がどんな失敗を起こすか呼び出し側から見えない、という点にある。Result 型で成功値と失敗値の両方を戻り値に表せば、起こりうるエラーが型に現れ、呼び出し側は失敗の処理を省けない。さらに、扱うのはドメインに関係するエラーに絞り、外部例外は Result を返す関数で包む。

  • 成功と失敗を Result 型で表し、シグネチャにエラーを明示する。
  • ドメインに関係するエラーだけを扱い、外部の例外は Result に変換して取り込む。

🛤️成功と失敗を2本の道として連結する

処理を成功の道と失敗の道の2本に見立て、bind と map で各ステップをつなぐ。

著者の筋道はこうである。各ステップが Result を返すと、ステップを単純に連結できない。そこで bind を「分岐を接ぐ部品」として使い、前のステップが成功なら次へ進め、失敗ならその先を飛ばして失敗を持ち越す。map は成功の道の値だけを変換し、失敗の道はそのまま通す。これにより、いったん失敗するとそれ以降の処理を短絡して失敗を運ぶ仕組みが、各ステップに分岐コードを書かずに実現できる。著者はこの構成を Railway Oriented Programming（鉄道指向プログラミング）と呼ぶ。

成功の道

処理が進むかぎりこの道を通る。map は成功の値だけを変換する。

⇄
失敗の道

いったん失敗するとこの道へ移り、以降の処理を飛ばして失敗を持ち越す。

  • bind は Result を返す関数を連結する部品で、失敗を後続へ持ち越す。
  • map は成功の道の値だけを変換し、失敗の道には触れない。
  • 計算式（computation expression）で連結を読みやすく書け、非同期（Async）の効果も加えられる。
🔎
なぜそう言えるか

失敗を持ち越す処理を bind と map に閉じ込めるため、各ステップは成功時の処理だけを書けばよく、エラー伝播のコードが各所に散らばらない。

📎
補足（本書外）

計算式（computation expression）は、bind などの連結を裏で行いつつ、逐次的なコードのように書ける F# の構文機能を指す。

第11章

シリアライズ

💡

この章を一言で

シリアライズと永続化を別の関心として分け、ドメイン型を外向けの DTO に変換してからシリアライズする。

🔄ドメイン型を DTO に変換してシリアライズする

ドメイン型を直接シリアライズせず、シリアライズ向けの単純な DTO に変換する。

著者の論拠は、ドメイン型は制約や判別共用体を含み、そのままでは一般的なシリアライズ形式に素直に乗らない、という点にある。シリアライズ専用の DTO へ一度変換すれば、ドメイン型は外部形式の都合に縛られず、変換コードがその橋渡しを一手に引き受ける。シリアライズはワークフローの端で行い、中核のドメインロジックから切り離す。

  • シリアライズ向けに設計した DTO を介してドメイン型と外部形式を橋渡しする。
  • シリアライズの処理はワークフローの端に置き、ドメインロジックと分離する。

第12章

永続化

💡

この章を一言で

永続化をワークフローの端に押し出し、コマンドとクエリを分け、各コンテキストが自分のデータの保存を所有する。

🧱永続化を端に押し出す

データの読み書きをワークフローの入口と出口に集め、中核のロジックを純粋に保つ。

著者の筋道は、保存処理を中核に混ぜると、ドメインロジックがデータベースに依存してテストしにくくなる、という点にある。読み込みをワークフローの始めに、書き込みを終わりに置けば、中間のロジックは I/O を含まない純粋な関数のまま保てる。これは第3部を通じた「I/O を端に」という方針を永続化に適用したものである。

  • 読み込みは入口、書き込みは出口に置く。
  • 中核のロジックを I/O から切り離し、純粋に保つ。

🔀コマンドとクエリを分け、データの所有を明確にする

状態を変える操作と読み取る操作を分け、各コンテキストが自分のデータ保存を所有する。

著者は、コマンド（変更）とクエリ（参照）を分離する（CQS）と、それぞれを独立して設計・最適化できると説く。さらに、境界づけられたコンテキストは自分のデータストレージを所有すべきだとする。コンテキストが他のコンテキストのデータに直接触れると境界が崩れるため、データの所有を分けることで第1部で引いた境界を保つ。文書データベース・関係データベースの扱いやトランザクションにも触れる。

  • コマンドとクエリを分離し、それぞれを独立して扱う。
  • 各コンテキストが自分のデータ保存を所有し、境界を保つ。
📎
補足（本書外）

CQS（コマンドクエリ分離）は、状態を変更する操作（コマンド）と、値を返すだけで状態を変えない操作（クエリ）を分けて設計する原則を指す。

Keywords

第13章

設計を進化させ、きれいに保つ

💡

この章を一言で

要件変更を型に反映すると、修正が必要な箇所をコンパイラが教えてくれる。型に支えられた設計は、変更を重ねても整合を保ちやすい。

🧭型の変更にコンパイラが追従させる

要件変更で型を変えると、影響を受ける箇所がコンパイルエラーとして洗い出される。

著者の筋道は、ドメインを型で表していれば、要件変更はまず型の変更として現れる、という点にある。型を変えると、その型に依存するすべての箇所がコンパイルを通らなくなり、修正すべき場所が機械的に列挙される。これにより、変更の影響範囲を人手で探す手間と漏れが減る。著者は具体的な変更（送料の追加、VIP 顧客への対応、プロモーションコード、営業時間の制約）を順に加え、この進め方を示す。

  • 要件変更を型の変更として表し、影響箇所をコンパイラに洗い出させる。
  • 送料・VIP・プロモコード・営業時間の追加を例に、変更への対応を示す。
🔎
なぜそう言えるか

ドメインが型で表されているため、型を変えるとそれに依存する箇所がコンパイルエラーになり、修正すべき場所が漏れなく可視化される。
