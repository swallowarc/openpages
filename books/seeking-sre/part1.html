第1章

SREにおけるコンテキストとコントロール

💡

この章を一言で

運用は、目標未達を罰する統制（control）より、判断の根拠を共有する文脈（context）を中心に据えるべきである。

🧭文脈を共有し、自律的な改善を促す

control は目標を外したチームに本番デプロイ権限の制限などの懲罰的措置を課す運用である。これに対し context は、判断に必要な情報を渡してチーム自身に改善させる運用を指す。著者は条件が許す限り後者を選ぶべきだとする。

著者は context を「ある要求や主張の背後にある根拠を、相手がよりよく理解できるようにする追加の関連情報」と定義する。この定義から、十分な情報を得たエンジニアは強制されなくてもより良い判断を下す、という前提が導かれる。Netflix ではマイクロサービス単位の可用性スコアカード（推移する可用性・目標値・下流依存の可用性）をチームに共有し、必要なときに協働して改善する運用をとっている。control 型の懲罰的措置をこれに対置し、文脈共有のほうが優れると結論づける。

  • context の中心は「根拠を理解させる情報」であり、命令や罰ではない。
  • Netflix はスコアカードを共有し、目標未達時も協働で改善する立場をとる。
  • 文脈駆動を支えるのはインフラ設計（イミュータブルなデプロイ、ガードレール）である。
❓
なぜそう言えるか

適切な情報があればエンジニアは良い判断を下す、という前提に立てば、自律的改善のほうが懲罰より持続的に可用性を高めるからである。ただし著者はこれを、組織が一定の失敗を許容でき、チームに協働の成熟がある場合の条件付きの主張として述べている。

第2章

サイトリライアビリティエンジニアの面接

💡

この章を一言で

プロジェクトの成否は技術ではなく適切な人材の採用で決まり、構造化された採用ファネルが面接官のバイアスを抑える。

🎯技術より人材、評価は標準化する

著者は「技術は適切な人材なしには何の役にも立たない」とし、採用の質を SRE 組織の成否の中心に据える。そのうえで、選考段階ごとに異なる能力を診断する構造化ファネルと、標準化された評価基準を置くことを主張する。

論拠は AOL・Google・YouTube・Dropbox での自身の採用経験である。採用は候補者・リクルーター・採用マネージャの最低3者で構成され、電話スクリーン・オンサイト・テイクホーム課題といった各段階がそれぞれ別の能力を測る。標準化と「ブラインドな履歴書レビュー」はバイアスを下げる手段として位置づけられる。前提には、標準化が組織の文脈を越えて採用品質を高め、SRE には従来のソフトウェアエンジニアと異なる評価軸が要る、という考えがある。

  • 採用は最低3者（候補者・リクルーター・採用マネージャ）で構成される。
  • 選考段階ごとに測る能力を変え、標準化した基準で評価する。
  • ブラインドな履歴書レビューなどでバイアスを低減する。

第3章

なるほど、SREチームを作りたいのですね

💡

この章を一言で

SRE を導入する前に「なぜ望むのか」という真の動機と、組織の文化的な準備が整っているかを問うべきである。

🤔動機を見極め、文化の準備を問う

著者は SRE 導入の動機を、流行への追随・障害への事後反応・戦略的投資といった類型で整理する。既存チームに「SRE」というラベルを貼るだけでは失敗し、SRE はラベルや用語の採用を超えた組織的コミットメントを要すると主張する。

この主張は、2014年以降に Google Cloud で100人を超えるリーダーと関わり、成功例と苦戦例を観察した経験を論拠とする。著者は「組織が工学的問題に感情で反応するなら、その組織は SRE の準備ができていない」と述べ、データ駆動で非難しない（blameless）文化を前提条件に置く。さらに成功には複数年の組織的投資、開発チームの運用への参加、経営層による SLO とエラーバジェットの支持が要るとし、100%未満の信頼性目標を受け入れて革新や速度とバランスさせることを条件に挙げる。

  • 導入動機を類型化し、それぞれの機会と落とし穴を見極める。
  • 工学的問題に感情で反応する組織は、まだ SRE の準備ができていない。
  • 成功には複数年の投資と経営層によるエラーバジェットの支持が要る。

第4章

インシデントのメトリクスを用いたSREの大規模な改善

💡

この章を一言で

信頼性投資は測定された指標でデータ駆動に優先順位付けし、修復負債を通常の開発プロセスに組み込むべきである。

📊障害を段階分解して測る

著者は「測定しなければ改善できない」という原則を掲げ、曖昧な「全体の信頼性」を、動かせるサブ指標に落とし込むことを主張する。障害のライフサイクルを段階に分解し、各段階を時間指標として測る。

規模が拡大すると「どの領域にどれだけ投資して信頼性を保つか」の議論が避けられなくなる。著者は Azure を規模・成長・多様性のケーススタディとし、障害を検知までの時間（TTD）・対応着手までの時間（TTE）・修復までの時間（TTF）に分解し、その合計を緩和までの時間（TTM）として捉える枠組みを示す。これにより、漠然とした信頼性が個別に改善できる指標へと変わる。さらに取りこぼした修復項目を「修復負債」として追跡し、標準の開発プロセスに組み込むことで、負債は安定でも信頼性が改善しないサービスを可視化できたとする。

  • 障害を TTD・TTE・TTF に分解し、合計を TTM として測る。
  • 自動検知は良い TTD の必要条件だが十分条件ではない。
  • 修復負債を追跡し、通常のエンジニアリング工程に統合する。

Keywords

インシデント指標TTD/TTE/TTF/TTM修復負債好循環Azure
第5章

サードパーティとの協力を円滑に進める重要性

💡

この章を一言で

外部ベンダー依存もサイトを落としうる重大リスクであり、社内と同じ SRE の規律を外部にも適用すべきである。

🔌外部依存にもSRE規律を対称に適用する

著者は DNS・CDN・APM・ストレージ・決済・メール・SSO/2FA・ログなど外部依存先を網羅し、適切に実装されなければそのいずれもがサイトを落としうると指摘する。ベンダーを「ブラックボックス」と諦めず、社内 SRE と同じ規律を適用すべきだと主張する。

導出は単一障害点の論理に立つ。外部依存はどれも障害の起点になりうるため、現状のベンダー管理が不十分なら、社内で使う SRE の規律をそのまま外部に対称適用できる、と考える。そのうえで Build か Buy かの判断を「重要度」と「ステークホルダー」の組み合わせで構造化し、重要度の判定を最初の一歩とする。ベンダー評価軸として、カスタマーサポート・価格・スケーラビリティ・柔軟性・ロードマップ・実績・実環境での検証を挙げる。

  • 外部依存先はいずれも単一障害点になりうる。
  • Build/Buy は重要度とステークホルダーの組み合わせで構造化して決める。
  • 評価軸はサポート・価格・拡張性・柔軟性・ロードマップ・実績など多面的。

Keywords

サードパーティ単一障害点Build/Buyベンダー評価
第6章

専任SREチームなしでSREの原則を適用する方法

💡

この章を一言で

少人数で多数のサービスを運用する中規模組織では、専任チームを置けなくても「精神としてのSRE」で持続可能な運用を実現できる。

⚖️運用を全員の仕事に再配置する

著者は、比較的少数のエンジニアが多様で多数の機能を開発・運用する中規模組織では、教科書どおりの SRE が適合しないとする。専任 SRE を持たなくても、原則の集合としての SRE は実践できると主張する。

SoundCloud はモノリスからマイクロサービスへ段階的に移行し、数百のサービスを少人数で運用してきた。元 Google SRE を雇ったが「教科書どおりの SRE」では機能しなかったという経験が起点になる。標準型のチーム構成はオンコールローテーションに必要な最小人数を満たせないため、著者は SRE を「チーム構造」ではなく「原則の集合」と捉え直す。具体的には、運用の自動化、適正な人数のオンコール、運用作業を50%未満に厳格に保つこと、メトリクスを至るところに置くこと、機能と安定性の自己調整である。「真の DevOps」では分離した dev/ops チームも役割も存在せず、運用責任は各エンジニアが負うとする。

  • 少人数×多サービスでは教科書型のチーム編成が成立しない。
  • 運用作業を50%未満に保つことが持続性の鍵になる。
  • 運用責任を全エンジニアの機能の一部に再配置する。

Keywords

中規模組織50%ルールSRE in spiritSoundCloud真のDevOps
第7章

SREのいないSRE：Spotifyのケーススタディ

💡

この章を一言で

Spotify は中央集権的な専任SREチームを持たないが、すべての活動にSRE原則を適用する能力でスケールしてきた。

🎵運用責任を開発チームへ移譲する

著者らは、Spotify には中央 SRE チームも SRE 専任チームも存在しないが、その時々のスケールに耐えてこられたのは原則をあらゆる活動に適用する能力によると述べる。この体制は最初から設計されたのではなく、スケーリングの痛みを通じて段階的に到達したものだと位置づける。

歴史的経路がこの主張を支える。ユーザー急増でピーク時の障害が多発し、100万ユーザー規模になっても手作業でデプロイしていた時期を経て、「システム所有者」「コアサービスの定義」「ゴールキーパー」といった役割を導入し、続く「Ops-in-Squads」期にデプロイや DNS を自動化して権限を開発チームへ移譲した。前提として、各 Squad が機能やユーザー体験を端から端まで所有し、インフラチームがセルフサービス型のインフラ製品を提供することが要る。一方で、横断的（cross-cutting）な関心事は中央の視点からしか扱えない領域として残るとも述べる。

  • 中央SREを持たず、運用責任を自律的な開発チーム（Squad）が所有する。
  • 体制は設計の産物ではなく、スケーリングの痛みを通じて段階的に到達した。
  • 横断的な関心事には、中央の視点でしか扱えない領域が残る。

第8章

大企業におけるSREの導入

💡

この章を一言で

SaaSスタートアップ流のSREはそのまま伝統的大企業に持ち込めず、マトリクス型の組織構造が新パラダイムの障害になる。

🏛️高頻度リリースが既存の組織枠組みと衝突する

著者の組織 Agilent Technologies は1930年代に遡るシリコンバレーの歴史を持つ。著者は、機能とプロダクトラインに二重で報告するマトリクス型の構造は、柔軟である一方で新しいパラダイムの導入を難しくすると指摘する。本章は SRE が必要だと理解しつつ実装方法に悩むマネージャー向けの実装ガイドだと位置づける。

論拠は、ハードウェア企業だった同社が SaaS を提供し始め、CI/CD で週2回ほどの高頻度リリースを行うようになったという変化である。高頻度リリースには従来とは別の運用モデルが必要で、伝統的な組織の枠組みやプロセスには収まらない。ここから、伝統的 IT・運用・サポートや品質・プロダクトエンジニアリングが互いに協調して動く必要があり、思考様式を組織横断で変えなければならない、という筋道が導かれる。

  • マトリクス型構造は柔軟だが、新パラダイム導入の障害にもなる。
  • SaaS の高頻度リリースは従来の運用モデル・組織プロセスに収まらない。
  • 本章はSRE導入の実装に悩むマネージャー向けのガイドである。

第9章

25ページでシステム管理者からSREへ

💡

この章を一言で

SREとシステム管理者を分ける根本特性は「視点」であり、SREは望ましい状態をビジネスニーズの観点で定義する。

👁️同じデータでも視点が変われば結論が変わる

著者は SRE を「多様な企業が大規模システムを効率的かつコスト効率よく運用・支援するための実践の集合」と定義する。そのうえで、SRE と従来のシステム管理者を分ける根本特性は「視点（point of view）」だと主張する。

システム管理者の視点はシステムがエラーを出さず過負荷にならないことに置かれるのに対し、SRE はシステムの望ましい状態をビジネスニーズの観点で定義する。著者は、同じメトリクスでも視点が違えば正反対の結論に至ると論じる。システム管理者には数ミリ秒のレイテンシ増加は大量のエラーに比べて重要に見えないかもしれないが、SRE にとっては「エラーが起きても、エンドユーザーに影響がなければサービスは健全」となる。技術的な完璧さよりユーザー影響というビジネス指標を測ることが重要だ、という導出である。

  • SRE は望ましいシステム状態をビジネスニーズの観点で定義する。
  • 同一のメトリクスでも、視点が違えば健全性の判断は逆になりうる。
  • 判断基準は技術的完璧さよりエンドユーザーへの影響を優先する。

第10章

大企業でSRE導入の道を開く方法

💡

この章を一言で

大企業には固有の構造的障壁があるが、ツール・スキル・組織のワークフローを同時に変革すれば SRE へ移行できる。

⛰️3つの層を同時に変革する

著者は、エンタープライズには小規模組織には存在しない構造的障壁があり、それが SRE 導入を阻むとする。それでも、障壁を特定して体系的に除去すれば移行は達成できると主張する。

本章は、大企業の運用組織の変革を支援するなかで得た知見の集成だと位置づけられる。著者は変革対象を、ツール（変更は実行しやすい）・スキル（育成は困難だが達成可能）・組織のワークフロー（最も難しい）の3層で整理する。これらを同時並行で変えなければ導入は成功しないとし、最難関である組織ワークフローの変革に焦点を当てて「道を開く」道筋を示す。すでに変革に着手した企業の知見が、他社の指針になるという前提に立つ。

  • 大企業の運用モデルは SRE のアプローチと根本的に異なる。
  • ツール・スキル・組織を同時に変革しないと導入は成功しない。
  • 最難関は組織のワークフロー変革であり、そこに焦点を当てる。

Keywords

構造的障壁ツール・スキル・組織ワークフロー変革
第11章

DevOpsの幅広い実践現場で活用されているSREのパターン

💡

この章を一言で

DevOpsが当たり前に使うパターンの多くはGoogleのSREで先駆けて生まれており、SREとDevOpsは相補的である。

🔗DevOpsの定番パターンの系譜をSREに遡る

本章は『The DevOps Handbook』からの編集抜粋である。著者は SRE と DevOps を相互排他ではなく相補的なものとし、現代の DevOps で当たり前に使われるパターンの多くが Google の SRE で先駆けて生まれたと述べる。

論拠として、Ben Treynor Sloss の「SRE とは、ソフトウェアエンジニアが運用組織を作ったときに起こることだ」という定義を引く。『The DevOps Handbook』の執筆調査で、多くのパターンが Google SRE 起源だと観察したという。本章は具体例として、Google における自動テストの誕生、ローンチ・引き継ぎの準備レビュー、共有ソースコードリポジトリの3パターンを挙げる。SRE を工学としての運用と定義したうえで、DevOps が好むパターンの系譜を SRE に遡らせ、両者は相補的だと結論づける。

  • SRE は「ソフトウェアエンジニアが運用組織を作ったときに起こること」。
  • 自動テスト・準備レビュー・共有リポジトリなどが SRE 由来として挙がる。
  • SRE と DevOps は対立せず相補的に働く。

第12章

DevOpsとSRE：コミュニティからの声

💡

この章を一言で

DevOpsとSREの関係は決着のついていない問いであり、確定的な答えより多様な現場の声が有益である。

🗣️未決の問いに、現場の声を並べる

本章は編者が企画・編集したコミュニティ寄稿の集成であり、単一著者の論考ではない。著者は DevOps と SRE の関係を「決着のついていない問い」と捉え、確定的な結論を出さず、多様な視点を並べることを選ぶ。

方法は、専門家にウェブサイトで2段落の回答を募り、世界各国から寄せられた多数の寄稿を集めるというものである。回答は「ごく軽く校正しただけ」で生の声を保って掲載される。クラウドソースされた現場の声のほうが、専門家による一方的な断定よりも、進化中の業界論争に対する洞察を与える、という前提に立つ。両者の類似と相違、組織内で共存可能かという開かれた問いを設定し、ニュアンスと不一致をそのまま可視化する。

  • DevOpsとSREの関係を、唯一の答えのない問いとして扱う。
  • 世界各国の寄稿を、ほぼ無編集で並列に提示する。
  • 確定的結論より、視点の多様性と不一致の可視化を重視する。

🛠️サイロを崩し、作り手に運用を持たせる

本章は Pedro Canahuati への編者のインタビューである。著者は、運用上の問題はソフトウェアで解決すべきであり、ソフトウェアを実際に作るエンジニアこそ本番でそれを運用する最適な人材だと主張する。開発・QA・リリース・運用を別チームに分ける従来の慣行は、問題のあるハンドオフを生むとする。

歴史的前提として、初期のソフト開発では開発者が自分のコードのデバッグも、さらにハードウェア問題までも自ら扱っていた。リモートシステムや大規模データセンターの登場で開発・QA・デプロイ・運用が分離したが、著者はその分業が機能する条件を「QA と運用が十分な技術力を持ち、コミュニケーション経路が健全であること」とする。その条件が崩れると本番修正が滞り、責任が元の開発者へ跳ね返る。境界を崩してソフトウェアエンジニアに運用責任を持たせる Facebook の Production Engineering モデルが、サイロ由来の非効率を解消する、という筋道である。

  • 運用上の問題はソフトウェアで解決し、作り手が運用する。
  • 分業は、技術力とコミュニケーションが健全なうちしか機能しない。
  • 境界を崩すことでハンドオフ由来の非効率を取り除く。
